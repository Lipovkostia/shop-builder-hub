# Интеграция с Авито — как работает и что можно улучшить

## Часть 1. Ответы на вопросы (как реально устроен Avito API)

### Можно ли обновить 1 товар (только описание/фото)?
**Нет, напрямую через core API нельзя.** Это ключевое ограничение Авито, не наше:
- `api.avito.ru/core/v1/items/{id}` — только **чтение** объявлений.
- Обновление полей объявления (title, description, images, params) через прямой REST **не поддерживается**. В коде нашей функции `avito-api` это честно отражено: `update_item` возвращает «Обновление через API ограничено».
- **Единственный способ массово править объявления — Autoload (XML/JSON-фид).** Авито забирает фид по расписанию, сравнивает по `Id` (наш внешний ID товара) и обновляет описание/фото/цену. Это как раз то, что у нас уже отдаёт `avito-feed`.
- Цену **одного** объявления можно поменять отдельным запросом `POST /core/v1/accounts/{user_id}/items/{item_id}/prices` — это единственное «точечное» обновление.

**Вывод:** «обновить одно описание/фото» = поправить товар у нас → следующий проход автозагрузки Авито подтянет изменения (обычно 1–4 часа). Мгновенного REST-апдейта нет.

### Можно ли создавать новые объявления отсюда?
**Да, но только через Autoload-фид** (`POST` объявлений напрямую тоже отсутствует). Мы добавляем новый товар → он попадает в `avito-feed` XML → Авито создаёт объявление при следующей выгрузке. У нас фид уже работает.

### Категории Авито — можно ли вытащить выпадающий список?
**Частично.** У Авито есть:
- Справочник категорий и обязательных параметров — публикуется как XSD/JSON schema для autoload (`autoload.avito.ru/format/...`). Это **статический справочник** (~150 категорий + сотни подкатегорий), его проще закешировать у нас в таблице, чем дёргать по API.
- Прямого core-эндпоинта «список всех категорий» нет, но есть валидатор фида, который вернёт ошибку, если категория/атрибут указаны неверно.

**Решение:** один раз импортировать справочник Авито в таблицу `avito_categories` (id, name, parent_id, required_params jsonb) и показывать его как красивый каскадный селект с поиском. Поддерживать обновление кнопкой «Обновить справочник».

### Можно ли вытаскивать ошибки модерации и чинить их через ИИ?
**Да, это самая сильная часть API.** Endpoint:
```
GET /autoload/v2/reports          — список отчётов автозагрузки
GET /autoload/v2/reports/{id}/items?status=error  — объявления с ошибками
```
Возвращает: `avito_id`, `title`, `error_code`, `error_message` (например «Категория не соответствует товару», «Запрещённый контакт в описании»). Этого достаточно, чтобы ИИ автоматически:
- сменил категорию (выбрал ближайшую валидную из справочника),
- переписал описание (убрал ссылки/телефоны/капс),
- перегенерил/обрезал картинку,
- отправил исправленный товар в следующий фид.

## Часть 2. Что предлагаю построить

### A. Система «уникальных дублей» одного товара под несколько объявлений Авито

Идея: один товар в нашем ассортименте → N «вариантов объявления» для Авито (разные тексты/фото/города/категории), при этом:
- В основном каталоге дубль **не появляется**.
- Видна связь «откуда сделан дубль» — удобное управление.

**Структура данных** (новая таблица, без изменения `products`):
```
avito_listing_variants
  id, store_id
  source_product_id  → products.id  (родитель, остаётся единственным в ассортименте)
  variant_label      ("Москва / спецпредложение")
  title, description, images[], price
  avito_category_id, avito_params jsonb
  city, address_id
  status             (draft | queued | published | moderation_error | archived)
  avito_item_id      (после публикации)
  last_error         (текст ошибки модерации)
  unique_seed        (для AI-уникализации, чтобы не считалось копией)
  created_at, updated_at
```
- В фид `avito-feed` подмешиваем строки из `avito_listing_variants` **в дополнение** к обычным товарам.
- В ассортименте товар остаётся один; у него появляется бейдж «Варианты для Авито: 3» → клик открывает менеджер вариантов.

### B. Менеджер вариантов (UI)

Внутри уже существующего `AvitoAiDescriptionWorkspace` добавить вкладку **«Дубли объявления»**:
- Кнопка «Создать дубль» → дублирует текущий черновик (текст + выбранные фото) как новый вариант.
- Список вариантов в виде карточек: миниатюра, заголовок, статус, город, категория.
- Кнопка «🪄 Уникализировать через ИИ» — массово прогоняет варианты через `ai-avito-description` с разными seed (другие синонимы, порядок блоков, эмодзи, перестановка фото) — чтобы Авито-антифрод не склеил.
- Видно «Родитель: SKU XXX» в каждом варианте + обратная ссылка из карточки товара.

### C. Каскадный селектор категорий Авито

- Новая таблица `avito_categories` (id, parent_id, name, path, required_attributes jsonb).
- Edge function `avito-categories-sync` — разово загружает справочник из autoload-схемы Авито (XSD → JSON), потом обновляется по кнопке.
- В форме варианта: компонент `<AvitoCategoryPicker>` — двух/трёхуровневый Combobox с поиском («Молочные → Сыры → Твёрдые сыры»), показывает обязательные атрибуты выбранной категории и валидирует их перед сохранением.

### D. Панель «Ошибки модерации» с авто-починкой ИИ

Новая вкладка **«Модерация»** в Avito-разделе:
- Edge action `fetch_moderation_errors` → дергает `/autoload/v2/reports/...` и сохраняет в новую таблицу `avito_moderation_issues` (avito_id, variant_id, error_code, error_message, raw, fixed_at).
- Список ошибок: фильтр по типу (категория / контакты / фото / описание).
- Кнопка **«Починить через ИИ»** на каждой строке:
  - если ошибка `wrong_category` — ИИ предлагает 3 ближайшие валидные категории из справочника, одним кликом меняем;
  - если ошибка `forbidden_content` — `ai-avito-description` переписывает текст, очищая запрещёнку;
  - если ошибка `image_*` — отправляем в `avito-image-edit` (рекадрировать/убрать водяной знак).
- Кнопка «Починить всё» — батчем по списку.
- После починки вариант помечается `queued`, попадает в следующий фид; через 1–4 часа Авито переотправляет в модерацию.

### E. Точечное обновление цены (бонус)

Для цены — единственного поля, которое API правит мгновенно — добавить кнопку «Опубликовать цену сейчас» рядом с полем цены в варианте → дергаем `POST /core/v1/accounts/{user_id}/items/{item_id}/prices`. Полезно для распродаж.

## Технические детали

**Новые таблицы (миграция):**
- `avito_listing_variants` (см. выше) + RLS по store_owner.
- `avito_categories` (общий справочник, читаемый всем authenticated, пишет только сервис).
- `avito_moderation_issues` + RLS по store_owner.

**Новые/изменённые edge functions:**
- `avito-api`: добавить actions `fetch_moderation_reports`, `update_price`, `sync_categories`.
- `avito-feed`: подмешивать строки `avito_listing_variants` (поле `Id = variant_<uuid>`), чтобы Авито различал их как отдельные объявления.
- `ai-avito-description`: расширить — режим `uniquify_variant` (на входе текст + seed, на выходе перефразированный текст с гарантией непохожести по shingle-overlap).
- Новая `ai-avito-fix-moderation`: на входе ошибка + текущий вариант → на выходе patch (новая категория / новый текст / новые картинки).

**UI:**
- Новые вкладки в `AvitoSection` → `Дубли`, `Категории`, `Модерация`.
- Расширить `AvitoAiDescriptionWorkspace` менеджером вариантов.
- Новый `AvitoCategoryPicker` (каскадный комбобокс).
- Новый `AvitoModerationPanel` со списком ошибок и кнопками авто-починки.

**Карточка товара в ассортименте:**
- Бейдж «Авито: N вариантов» → клик ведёт в менеджер.

## Что НЕ войдёт (честные ограничения Авито)

- Мгновенного REST-обновления description/images у одного объявления **не существует**. Минимум 1–4 часа лага через autoload. Цена — исключение.
- Создание объявления тоже только через фид (не «по кнопке мгновенно»).
- Список категорий через core API недоступен — используем закешированный справочник.

Если согласуете план — реализую поэтапно: сначала **A+B (дубли и уникализация)**, затем **C (категории)**, затем **D (модерация + авто-починка)**.
